## 1. Honest verdict on publication / ship readiness

The methods record (`ihc.md`) and the statistical core are, on their own, unusually
rigorous and self-critical — this project genuinely practices "honest reporting of
failures." The cross-K / DCLF / intensity-reweighted-inhomogeneous machinery is
correctly implemented, cross-validated against `spatstat` to floating point (§16),
its edge-correction decision is empirically justified (§17), and its headline §15.4
calibration (shared-preference size 0.032 at bw=75 µm) **reproduces against the
current code** (I re-ran it). Deps are pinned with a real rationale; outputs carry a
provenance block; the JSON keys and most UI copy correctly avoid co-expression
language; the Restained tab is genuinely isolated and fail-closed.

**But the project is not ready for external/publication use as a CD8/TIM-3 tool**,
for one structural reason and several traceability reasons:

- **The entire §18–20 landmark-certification chain is offline-only and is not wired
  into the shipping app or CLI spatial pipeline** (A1/B1). What a biologist actually
  runs in the Spatial tab uses the *legacy* automated registration cascade plus a
  feature-matching QC gate that **§18.4 itself proves is unreliable on exactly these
  serial sections.** The honest conclusion the docs reach ("no CD8/TIM-3 pair has
  ever been CERTIFIED; Phase B must not run on 052526") is enforced only in
  validation scripts; the app will compute and display a green "associated (robust)"
  verdict on those same uncertified pairs. This is the gap between the documented
  honesty standard and the product.
- A small set of **stated numbers/schemas are stale or unsupported**: the "~90%
  segmentation agreement" claim is labelled *unsupported by its own validator* (A2);
  `ihc.md` §6–§7 and the production docstrings describe the **retired** three-null
  design as current (A3); §9b's real-data table validates the **retired** CSR null,
  not the production primary (A4); and a "Do NOT ship this null" log ships next to
  the cited "SHIP" log (A5).

"Defensible as-is for internal/method-development use" — yes. "Defensible for an
external CD8/TIM-3 biological claim" — no, and the docs themselves say the biology is
not established; the problem is that the *app* does not enforce that discipline.

## 2. PASS A — peer-reviewer findings

### A1 — Certification chain is not wired into the production path *(Tier 1: correctness/process)*
**Claim.** §18.5/§19/§20.6–20.7 present a landmark-driven, 4-state certification
(CERTIFIED / LOCALLY_CERTIFIED / DEFORMED / NOT_CERTIFIABLE), a fail-closed QC gate,
a perturbation-sensitivity test, and the headline honesty that "Phase B must not run
on uncertified pairs."

**Evidence it is not in production.**
- `serial_registration.py` (all of `landmark_register_and_verify`, `loo_tre`,
  `registration_perturbation_sensitivity`, the 4-state verdict) is imported **only**
  by `validation/*` (`phase_a_*.py`, `validate_hyreco.py`, `validate_anhir_landmarks.py`,
  `validate_phase_b_certified.py`). Grep of `*.py webui/*.py app.py` for
  `serial_registration` / `landmark_register` / `certified_transform` /
  `registration_perturbation`: **no production hit.**
- The app/CLI Spatial path is `run_pipeline.py:1072` → `spatial.run_spatial_association`
  → `registration.compute_registration` (legacy SimpleITK→ORB/SIFT→phase→identity
  cascade) + `run_pipeline.evaluate_registration_qc` (`run_pipeline.py:865`) consuming
  `registration.compute_registration_qc` (`registration.py:426`).
- The production QC residual is computed from **feature-point correspondences in the
  hematoxylin channels** (`registration.py:493`, `_qc_match_points`). §18.1 and §18.4
  state plainly that this class of metric is untrustworthy on these tissues:
  "Corner/keypoint landmarks … do not correspond across serial sections; mutual-NN
  matches are random," and the broader §18.4 conclusion that *no automated metric*
  reliably measures µm-level TRE on FOV-crop serial sections of this tissue.
- `grep -ci "landmark\|certif" webui/index.html` = **0**. The shipping UI has no
  certification concept at all. The only landmark tool is generated to
  `validation/phase_a_qc/landmark_tool.html` — an offline artifact bound to the 7-pair
  052526 cohort, not part of `webui/`.

**Why it matters.** A pair that the §19 process would mark NOT_CERTIFIABLE/DEFORMED
can, in the app, pass the legacy feature-match gate (residual < 5 µm against whatever
correspondences the matcher finds — §18.5 round-2 documents *spurious* consistent
matches in dense lumen fields) and be rendered as `statistics_valid: true` with a
"robust" badge. The fail-closed legacy gate is conservative in the common case
(random matches → ~15 µm → invalid; identity → invalid), so the dominant failure is
*over-rejection*, not silent acceptance — but the path by which an **uncertified**
pair is presented as **valid** is open, and there is no UI signal that "valid here"
is a weaker statement than "CERTIFIED" in the docs.

**Minimal honest fix (before any external use).** Either (a) wire
`landmark_register_and_verify` + the 4-state verdict + perturbation test into the
Spatial tab (this is §19.6 "future work" — promote it), gating the statistics on a
real certification; or (b) if shipping the legacy path, add a prominent, blocking UI
disclaimer that the app's automated registration QC **is not §18–20 certification and
is known-unreliable on FOV-crop serial sections**, and stamp every spatial result
JSON with `certification: "not_performed"` so downstream consumers cannot mistake
`statistics_valid` for a certified pair.

### A2 — "~90% segmentation agreement" is unsupported by its own validator *(Tier 1/2)*
**Claim.** `ihc.md` §3 (line ~116): "Validated against manual counts at roughly
**~90% agreement** on the development cohort."

**Evidence.** `validation/validate_segmentation.py:52-54` states verbatim that the
user must produce manual GeoJSON ground truth and that aggregating it *"REPLACES the
unsupported '~90% agreement' claim."* No manual ground-truth file ships in the repo;
no saved output produces a 90% number. The validator requires `--manual
<image>_manual.geojson` inputs that are not present.

**Note.** The only real segmentation ground-truth in the repo is the Restained-tab
HNSCC nuclear masks (§21.3–21.6), which give F1 0.776–0.847 on *mIHC* tiles — a
different dataset and a different endpoint (nuclear segmentation, not CD8/TIM-3 DAB
quant classification). It does not back the "~90%" quant claim.

**Fix.** Delete the "~90%" number or replace it with a data-backed F1/κ from an actual
manual-count run; until such an artifact exists, mark the quantification core
"agreement unverified."

### A3 — §6 schema and §7 describe the RETIRED three-null design as current *(Tier 2: framing)*
**Claim.** `ihc.md` §6 result schema shows `"primary_null": "inhomogeneous"` with
`nulls: {homogeneous, inhomogeneous (+bandwidth_sensitivity), toroidal}` (lines
~228–243); §7 "Three null models (`cross_k_all_nulls`, n=1000 each)" describes
homogeneous+inhomogeneous+toroidal as the production set and says "the inhomogeneous
(primary) null's curves are also copied to the result's top level."

**Evidence this is stale.** Production default is
`cross_k_all_nulls(..., nulls=("reweighted","homogeneous"))` (`spatial_stats.py:941`);
`primary_null` resolves to `"reweighted"` (`spatial_stats.py:1050`); the retired
`inhomogeneous`/`toroidal` nulls "never gate the verdict" (`spatial_stats.py:955-958`).
§15.3 explicitly retires them. So a reviewer reading §6–§7 (the *primary* methods
sections) gets a description that §15 — nine sections later — overturns. The stale
description is echoed in the production code's own docstring
(`spatial.py:478-483`, "THREE null models … homogeneous CSR + inhomogeneous (Kinhom)
+ toroidal shift") and a UI comment (`webui/index.html:1741`, "Robustness across the
THREE null models").

**Fix.** Update §6 schema and §7 to the reweighted-primary + CSR-baseline design with a
forward pointer to §15; correct the `spatial.py` docstring and the `index.html`
comment. (Cosmetic, but in a paper-grade record the *main* statistics section should
not describe a retired design as current.)

### A4 — §9b "real-data biological validation" exercises the retired CSR null *(Tier 2/3)*
**Claim.** §9b headline table (CD8↔CD4 86% association, CD8↔tumour 58% segregation,
CD8↔Treg 68%) presented under "Real-data biological validation."

**Evidence.** `validation/validate_real_data.py` imports and calls **`cross_k_null`**
(the legacy single homogeneous-CSR null), `validate_real_data.py:63,138,221` — not the
production primary (`cross_k_inhom_reweighted_test`). §15.7 honestly re-runs through
the production primary and reports materially different, lower numbers (CD8↔CD4 →
**50% robust**; CD8↔tumour segregation "largely disappears"). §9 is not annotated to
say its table uses a retired null; a reader of §9 alone would carry the 86% number.

**Fix.** Add a one-line banner at §9b: "These figures use the retired homogeneous-CSR
null; the production-primary results are in §15.7." This is mostly already true in
spirit (§9 and §15.7 both exist) but the overturn is not flagged at the point of the
stale table.

### A5 — Stale contradictory calibration log; narrow shippable bandwidth window *(Tier 3: rigor-harness)*
**Claim.** §15.4: bw=75 µm with LOO gives shared-preference size **0.032**, uniform
0.064, power 1.000/0.992 → "winner," replacing the anti-conservative nulls.

**Evidence.**
- The cited artifact `validation/reweighted_null_output.txt` (NREAL=500, BW=50,60,75)
  shows exactly 0.032 / 0.064 / SHIP.
- **But `validation/_reweight_run.log`** — produced with the script's *current
  default* parameters (NREAL=300, BANDWIDTHS=50,75,100,150; cf.
  `validate_reweighted_null.py:31,33`) — shows bw=75 shared **0.170**, verdict
  "shared-leak," and a global **"⚠ NO bandwidth passes all three regimes. Do NOT ship
  this null as-is."** `_reweight_run2.log` (BW=30,40,50) is also a "Do NOT ship."
- Because the harness uses fixed seeds (`seed=1001/2002/...`) and evaluates each
  bandwidth independently, the two logs cannot both come from the same code with only
  NREAL differing — i.e. `_reweight_run.log` is from an **older code/harness version**
  and is now stale.
- **I re-ran the current code** (reduced NREAL=120/NPERM=99): bw=75 → shared **0.025**,
  uniform 0.083; bw=50 → shared 0.083, uniform 0.050. So the current code reproduces
  the *favorable* ~0.03 shared rate at bw=75 (the cited claim holds), and the 0.170 log
  is indeed stale.

**Two real residual concerns (not contradictions, but rigor gaps):**
1. Shipping a "Do NOT ship this null" log next to the cited "SHIP" log, with no
   annotation, makes provenance unauditable to a reviewer who cannot re-run.
2. The calibrated window is **a single bandwidth**: at bw=50 the shared rate leaks
   (0.083 > 0.07) and at bw=75 the uniform rate sits near the 0.07 ceiling (cited
   0.064; my reduced run 0.083). This is consistent with — and an instance of — the
   §15.5 disclosed limitation that the test is anti-conservative when architecture
   approaches the bandwidth. It is disclosed, but it means the production 75 µm
   constant is on a knife-edge.

**Fix.** Delete or clearly mark `_reweight_run.log` / `_reweight_run2.log` as
superseded; regenerate `reweighted_null_output.txt` from the current code at the
documented settings and cite that single artifact; keep the §15.5 caveat prominent.

### A6 — Anti-conservative failure mode disclosed but not guarded *(Tier 2)*
**Claim.** §15.5: "Bandwidth ≤ architecture scale is REQUIRED … false-`robust` rises
to 40–100% when the architecture scale drops to 45–70 px"; the method "assumes tissue
architecture is coarser than the 10–50 µm interaction band."

**Evidence.** Production uses a fixed `_REWEIGHT_BANDWIDTH_UM = 75.0`
(`spatial_stats.py:62`) regardless of the actual tissue's architecture scale, which is
**never measured** anywhere in the pipeline. There is no runtime check and no UI
warning that the "robust" verdict is only valid when architecture > ~75 µm. A
biologist analysing tissue with fine (cell-scale) architecture would get a confident
false-positive "robust association" with no signal that the core assumption is
violated.

**Fix.** Surface the assumption at result display (a one-line caveat on every "robust"
verdict), and ideally estimate a characteristic architecture scale (e.g. the KDE
intensity autocorrelation length) and downgrade/flag when it approaches the bandwidth.
At minimum, stamp the assumption into the result JSON.

### A7 — Provenance omits the primary null's actual bandwidth *(Tier 3)*
**Evidence.** `build_provenance` (`run_pipeline.py:752-759,785-792`) records
`kde_bandwidth_um` pulled from the **retired** `nulls.inhomogeneous` block (which is
not in the default null set), so it falls back to `_KDE_BANDWIDTH_UM = 50.0` — *not*
the reweighted primary's `_REWEIGHT_BANDWIDTH_UM = 75.0` that actually drove the
result. The fixed null `seed` (0) is also not stamped.

**Fix.** Stamp `reweight_bandwidth_um` and `null_seed` in `analysis_params`.

### A8 — Cross-pair FDR present for Restained, absent for the Spatial cohort *(Tier 3)*
**Evidence.** `cohort_multiple_comparison_correction` (`spatial_stats.py:1343`) is
applied across tiles in the Restained path (`webui/api.py:560-577`) and per-tile in
`restained_coexpression.py:598`. I found **no** application across pairs in the
Spatial-association cohort: each pair's DCLF p stands alone; running N pairs and
calling the p<0.05 ones "robust" has no family-wise/FDR control across the cohort.
Pass A item 3 asked whether the FDR harness is "present and applied" — present yes,
applied to the Spatial cohort **no**.

**Fix.** Apply BH across the per-pair DCLF p-values in the Spatial cohort summary (the
function already exists), or document that per-pair results are not cohort-corrected.

### A9 — Statistically defensible elements (earned credit, not assumed)
- **spatstat cross-validation (§16):** the estimator matches `Kcross.inhom`
  (`correction="none"`) to ~1e-10–1e-14 on synthetic *and* a real Schürch spot; the
  ~1e-3 Stage-A residual is a documented Gaussian-tail truncation. This is a genuine,
  well-scoped cross-check. The R script and saved output are present
  (`spatstat_crossval.R`, `spatstat_crossval_output.txt`).
- **Edge-correction decision (§17):** the translation-correction A/B at fixed seeds
  showing identical calibration to 3 d.p. is a correct symmetric-cancellation argument
  and is the right answer to "why no edge correction"; `validate_edge_correction.py`
  mirrors the production estimator.
- **DCLF construction** (`spatial_stats.py:350-434`): pooled-mean exchangeability,
  +1 correction, directional one-sided variants, band-limited to [10,50] µm — correct
  and matches the docs.
- **Intersection window, hard-core floor, neutral keys** (`CD8__TIM3`, not `CD8+TIM3+`):
  implemented as documented (`spatial.py:588-590`).
- **Negative-control honesty:** §15.7 explicitly states CD8–tumour is *not* a clean
  cell-scale negative control — exactly the kind of admission a reviewer wants.
- **MNN/co-expression legacy code** is clearly marked DEPRECATED/UNUSED
  (`spatial.py:18,358-369`) and is not called by `run_spatial_association`.

### A10 — Overclaim sweep result
The rename away from co-expression is **substantially complete** in the serial-section
path: result keys, the analysis-card UI blurb (`index.html:811`), README §"Run Spatial
Association", and the headline verdict strings all carry the "population-level, not
single-cell co-expression" caveat at the point of display. No code identifier, JSON
key, or figure label was found asserting CD8↔TIM-3 single-cell pairing for the serial
path. The remaining framing issues are A3/A4 (retired-design descriptions), not
co-expression overclaims. The Restained tab legitimately *does* make a same-cell
co-expression claim — appropriate, since it is same-section — and correctly brands it
exploratory (B4 below).

## 3. PASS B — biologist UI/UX findings

### B1 — App shows "robust" spatial verdicts with no certification and no §18 disclosure *(blocks-correct-use)*
The operator's view (the Spatial tab) renders a green "associated (robust)" badge
(`index.html:1838`) and a residual-µm "Registration passed QC" line
(`run_pipeline.py:943`) for any pair that clears the legacy feature-match gate. There
is **no landmark step, no 4-state verdict, and no statement** that this QC is the
automated metric §18.4 found unreliable on these tissues. A pathologist cannot tell
that "valid" here is a much weaker claim than the "CERTIFIED" the documentation builds
its entire credibility on. This is the Pass-A A1 gap seen from the operator's seat and
is the single most important misuse risk. **Fix:** as A1 — in-app certification, or a
blocking banner that the app's registration QC is not certification and must not back a
reported finding.

### B2 — "robust across the structure-preserving null models" copy is now false *(causes-misinterpretation)*
The headline interpretation string says results are "robust across the
structure-preserving null models" (`index.html:1808`) and a comment/section still
references "THREE null models" (`index.html:1741`). Production now runs **one**
calibrated primary (reweighted) + a CSR baseline; the toroidal/resampling structure-
preserving nulls are retired. A biologist reads "robust across [multiple] structure-
preserving nulls" as independent corroboration that no longer happens. The "Significant
/ calibrated null" metric label (`index.html:1667`) is correct; the prose around it is
stale. **Fix:** reword to "significant under the calibrated reweighted inhomogeneous
cross-K (size-controlled against shared tissue preference); homogeneous CSR shown only
as a shared-preference flag."

### B3 — Strong misuse-prevention affordances (earned credit)
- `csr_only` is rendered as a **red** "⚠ Significant ONLY under the weak CSR baseline —
  shared-preference artifact" (`index.html:1747,1809`), never as a pass.
- Registration-invalid pairs get a red banner "⚠ Registration failed QC — spatial
  statistics are unreliable" and the interpretation explicitly says the call "must NOT
  be reported as a finding" (`index.html:1735,1806`); `statistics_valid` is propagated
  (`run_pipeline.py:1187,1200`).
- The "not single-cell co-expression / requires multiplex" caveat appears at the
  analysis card, i.e. at the point of use (`index.html:811`).
These are exactly the guardrails the brief asks for and they are well done — the
weakness is B1/B2 (the registration-trust story), not the verdict presentation.

### B4 — Restained tab: §21.6 correspondence lesson is prose, not a guardrail *(causes-misinterpretation)*
The Restained run is correctly fail-closed on **dimension equality**
(`restained_coexpression.py:140-151`, raises with the offending dimensions) and on
**mandatory AEC thresholds** (UI placeholder "required" + non-null validation,
`restained_coexpression.js:73,77,82,158-159`; backend required-keys
`restained_coexpression.py:558-561`; segmentation DAB forced to 999 so no DAB default
leaks, `:415`). Two "Interpretation boundary" caveats correctly brand co-expression
exploratory (`restained_coexpression.js:73,196`). **However**, §21.6 demonstrated that
*equal dimensions ≠ shared coordinates*: tile `Case2_S3_1_1` produced an extreme false
double-positive signal (BH q = 5.46e-31) purely from grossly non-corresponding tissue
content that nonetheless had equal dimensions. The **only** automated correspondence
guard in the code is the dimension check, which §21.6 itself proves is insufficient.
The §21.6 zero-lag DAPI↔hematoxylin structural-correlation diagnostic exists *in the
all-image validation harness* (`validate_hnscc_restained_all.py`) but is **not** a gate
in the production run path. A biologist can therefore get a magenta double-positive
overlay + Fisher p on non-corresponding tiles with no automated flag. **Fix:** promote
the hematoxylin-vs-AEC (or expected-mask) structural-correlation check into the run as
a fail/warn gate, and add a UI caveat at result display that equal dimensions and
same-section provenance do **not** verify shared cell coordinates.

### B5 — `learn.md` is good but §20 is stale *(polish / minor misinterpretation)*
`learn.md` is genuinely plain-language and its §10 correctly tells the "we were fooled
85–87%, here is the fix" story matching `ihc.md` §15. But §20 ("Where things stand")
lists "three null models with a robustness verdict" as "Built and working"
(`learn.md:684`) — the same retired-design drift as A3/B2. **Fix:** update `learn.md`
§20 to the reweighted-primary + CSR-baseline design.

### B6 — Comprehension of plots/terms (acceptable, with gaps)
The result card provides plain-language interpretation of the verdict, the band, and
the peak radius, and figure captions name the L−r/envelope plot
(`index.html:1808,1832`). DCLF/L−r/envelope are not defined inline in the UI, but the
analysis-card blurb and `learn.md` cover them; for the target operator this is borderline
acceptable. Pixel-size/scale-bar calibration has a dedicated UI card
(`index.html:495,657-659`). The membrane/cytoplasm-ring toggle and DAB-vs-AEC
distinction are surfaced (Restained warn box explicitly says "These are not DAB
thresholds"). No blocking comprehension gap found beyond B1/B2.

### B7 — Landmark tool is unreachable in normal use *(friction, ties to B1)*
The blinded landmark tool (`validation/phase_a_qc/landmark_tool.html`, Blind=ON by
default — good, `make_landmark_tool.py:118,178`) is generated by an offline script and
bound to the 7-pair 052526 cohort. It is not discoverable from the desktop app. If
landmark certification is the intended gate (and the docs say it is), the operator has
no in-app way to perform it. **Fix:** as §19.6 future work — bring the picker into the
Spatial tab.
